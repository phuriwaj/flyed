# RB — Enquiry 429 storm

## Overview

`POST /api/enquiry` is rate-limited to **5 requests per 60-second sliding
window per client IP** (see
`/home/phurix/projects/flyed/src/pages/api/enquiry.ts:36-41` and
`/home/phurix/projects/flyed/src/lib/rate-limit.ts:30-52`). When a single IP
hits the cap, the handler returns `HTTP 429 { ok: false, error: "Rate limit
exceeded" }` with a `Retry-After` header.

A "storm" is when the 429 rate spikes above baseline — either because a real
user is hammering the form, or because `RATE_LIMIT_KV` has been evicted (see
`RB-rate-limit-kv-eviction.md` — eviction does NOT itself cause 429s, but
eviction immediately after a burst can produce _intermittent_ 429s that look
storm-like).

**User-visible impact:** users see "Something went wrong. Please try again." on
step 5 of the enquiry form (`EnquiryForm.tsx:147-152`). The form does not
specifically distinguish 429 from a generic HTTP error.

**Resolved means:** 429 rate returns to baseline (< 1/minute on average) and
no legitimate user is blocked.

**Severity guidance:** default **P3** (no customer-visible degradation for
most users). Escalate to **P2** if (a) more than 100 distinct IPs are 429-ing
in a 5-minute window, or (b) a paying-customer school reports they cannot
submit the form.

## Prerequisites and access

- Cloudflare account with `Workers` edit permission on the `flyed` project.
- `wrangler` CLI authenticated (`wrangler whoami` should show your account).
- Access to `/home/phurix/projects/flyed/.lighthouseci/`-hosted logs OR
  `wrangler tail` from a workstation with the Workers editor role.

## Diagnosis

> **Decision path.** Run the checks in order; each one rules out a category of
> cause before you go deeper.

### 1. Check recent changes (events-first)

```bash
git log --oneline -20 origin/main
wrangler deployments list --name=flyed
```

Most 429 storms start with either a deploy that changed the rate-limit
parameters or a code-deploy that unintentionally introduced a retry loop on
the client. Look for changes in
`/home/phurix/projects/flyed/src/pages/api/enquiry.ts` lines 36–48 or in
`/home/phurix/projects/flyed/src/lib/rate-limit.ts`.

### 2. Check the live 429 rate

```bash
wrangler tail --name=flyed --format=json | grep -E '"status":429' | head -20
```

Healthy: a handful of 429s per hour from retry-happy users. Storm:
double-digit 429s per minute, often from one or two IP ranges.

### 3. Isolate the source IPs

In the Workers Logs UI (Dashboard → Workers → flyed → Logs), filter by
status 429 and group by `clientIP`. One IP dominating = single source.
Distributed IPs = bot/scraper behavior or shared egress.

### 4. Confirm `RATE_LIMIT_KV` is bound

```bash
wrangler kv:namespace list --namespace-id RATE_LIMIT_KV
```

If the namespace is missing or the binding is not in
`/home/phurix/projects/flyed/wrangler.jsonc:18-21`, the rate limiter
**fails open** (`src/lib/rate-limit.ts:32-34`) and 429s are NOT generated by
KV — they would have to come from a different layer. If you see 429s and
the binding is missing, your 429s are coming from a Cloudflare layer
(WAF/bot-management), not from the application. Re-run step 3 with
WAF-aware tooling.

### 5. Check for client-side retry loops

If step 3 shows one IP but step 2's volume is high, look at the request
shape. A retry loop typically has the same `enquiryId`-less payloads
hitting `/api/enquiry` dozens of times per second from one IP. The fix is
in the client (`EnquiryForm.tsx:116-156`), not the Worker.

## Mitigation

### A. Single abusive IP

1. Add a Cloudflare WAF rule: IP Access Rule → Block (or Challenge) for the
   source IP. Use the smallest TTL possible (challenge first, block if it
   continues).
2. **Verification:** `wrangler tail` shows no further 429s from that IP.
3. Add an entry to the team's incident log (date, IP, duration, cause).

### B. Distributed burst (scraper / botnet)

1. Enable Cloudflare Bot Fight Mode or Super Bot Fight Mode at Security →
   Bots. This adds an interstitial before the Worker sees the request.
2. Verify rate-limit parameters are unchanged
   (`/home/phurix/projects/flyed/src/pages/api/enquiry.ts:39-40` —
   `max: 5, windowMs: 60_000`). Do NOT loosen them during the incident;
   doing so turns a P3 into a P1 (form abuse flooding the email pipeline).
3. **Verification:** 429 rate drops back to baseline within 10 minutes.

### C. Legitimate user / paying customer

1. Confirm the customer by cross-referencing the form's `schoolName`,
   `email`, or `phone` fields in the Worker logs.
2. If confirmed, escalate to the **engineering lead** and temporarily raise
   the limit by editing `max` and `windowMs` in
   `/home/phurix/projects/flyed/src/pages/api/enquiry.ts:39-40`. The change
   ships on the next deploy.
3. **Verification:** the customer confirms they can submit the form.

> **Destructive step warning:** Do NOT disable the rate limiter entirely by
> removing the call. A flood without a rate limit will exhaust the
> `RESEND_API_KEY` quota and burn through Resend credits.

## Verification

You know the storm is over when **all** of these are true:

- `wrangler tail` shows 429 rate back to baseline (< 5/hour).
- No customer tickets in the last 30 minutes.
- `/api/enquiry` error rate (any 4xx/5xx) is below 0.5% of submissions.

## Escalation

- **Time-box:** if not mitigated within 30 minutes, escalate to the
  engineering lead.
- **If a paying customer is blocked:** immediately escalate to the founder
  on call. Don't wait 30 minutes.
- **What to hand over:** the diagnosis output above (the IP list, the
  recent change history, the rate observed), plus your `wrangler tail`
  output as an attached log file.

## Post-incident

- Update the channel: "incident resolved, root cause: <single IP|botnet|client
  retry loop>, mitigation: <WAF rule|bot mode|rate-limit bump>, lead contact:
  <name or none>".
- Open a postmortem if duration > 1 hour or > 100 users affected.
- File a fix ticket if a client retry loop was the cause (it should not
  retry on 429 without backoff).
- **UPDATE THIS RUNBOOK** if any step was missing or wrong. Runbooks that
  drift from reality are worse than no runbook.
